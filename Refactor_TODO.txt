Ja — hier ist die konkrete Refactoring-Roadmap in einem einzigen Block.

REFORM-ZIEL
- index.js soll nur noch Composition Root / Startpunkt sein
- Fachlogik soll pro Domäne gekapselt werden
- Frontend-State und API-Orchestrierung sollen getrennt werden
- Dev-Workflow soll schneller und wartbarer werden

==================================================
PHASE 1 — SOFORTMASSNAHMEN MIT HOHEM HEBEL
==================================================

1. index.js auf Bootstrapping reduzieren
Warum:
- aktuell ist index.js zentraler Verdrahtungspunkt für App, Middleware, Session, DB, Repositories, Services, Controller und Routes
- das erschwert Tests, Orientierung und spätere Änderungen

Zielzustand:
- index.js startet nur noch den Server
- neue Dateien:
  - app/create-app.js
  - app/create-dependencies.js
  - app/config.js
  - app/create-db.js

Konkreter Schnitt:
- config.js:
  - PORT
  - NODE_ENV
  - SESSION_SECRET
  - DATA_DIR
  - DB_FILE
  - ENABLE_DEMO_DATA
  - TRUST_PROXY
- create-db.js:
  - DB öffnen
  - Pragmas setzen
  - DB-Objekt zurückgeben
- create-dependencies.js:
  - Repositories erzeugen
  - Domain Services erzeugen
  - Application Services erzeugen
  - Controller erzeugen
- create-app.js:
  - Express-App erzeugen
  - Middleware registrieren
  - Routes mounten
  - Error Handler anhängen
- index.js:
  - Runtime erzeugen
  - app.listen(...)

Nutzen:
- klarer Startpunkt
- leichtere Testbarkeit
- bessere Lesbarkeit
- weniger Kopplung

Risiko:
- niedrig bis mittel
- wichtig: nur verschieben, nicht gleichzeitig Logik ändern

--------------------------------------------------

2. Middleware- und Security-Setup bündeln
Warum:
- Security-Header, JSON-Parser, Session, Static Serving und Error Handling sollten nicht im Hauptfile verstreut sein

Zielzustand:
- middleware/register-core-middleware.js
- middleware/register-security-middleware.js

Enthalten:
- trust proxy
- static middleware
- json/urlencoded parser
- security header
- rate limiting
- session middleware

Nutzen:
- zentrale Übersicht
- klarer Unterschied zwischen Core- und Security-Middleware
- spätere Erweiterungen werden sauber einsortiert

--------------------------------------------------

3. Session-Konfiguration kapseln
Warum:
- Session-Handling sollte austauschbar sein
- ein externer Store kann später kommen, aber erst nach sauberer Kapselung

Zielzustand:
- sessions/create-session-middleware.js

Enthält:
- cookie config
- sameSite
- secure nur in production
- httpOnly
- session secret
- optionaler Store als austauschbare Abhängigkeit

Nicht zuerst machen:
- nicht sofort Redis einführen
- erst abstrahieren, dann Infrastruktur ändern

==================================================
PHASE 2 — BACKEND FACHLICH STABILISIEREN
==================================================

4. Von technischen Layern zu Feature-Modulen wechseln
Warum:
- aktuell gibt es bereits routes, services, repositories, controllers
- mittelfristig ist eine feature-basierte Struktur für dieses Projekt passender

Zielzustand:
- modules/auth/
- modules/report/
- modules/admin/
- modules/grades/

Beispiel:
modules/
  report/
    report-routes.js
    report-controller.js
    report-service.js
    report-domain-service.js
    report-repository.js
    report-schema.js

Warum das besser ist:
- alle report-bezogenen Teile liegen beisammen
- geringere Suchkosten
- weniger Querschnittschaos
- klarere Eigentümerschaft pro Bereich

Reihenfolge:
1. report
2. admin
3. grades
4. auth

Warum report zuerst:
- vermutlich meiste Domänenlogik
- meiste Änderungsintensität
- höchste Relevanz für Kernprodukt

--------------------------------------------------

5. Domain Services klar von Application Services trennen
Regel:
- Controller = HTTP rein/raus
- Application Service = Use Case, Orchestrierung, Rechtekontext
- Domain Service = Fachregeln
- Repository = Datenzugriff

Warnsignal:
Wenn ein Service gleichzeitig
- Rollen prüft
- DB aufruft
- Daten normalisiert
- Export rendert
- HTTP-nahe Reaktionen steuert
dann ist er zu breit

Konkrete Ziele:
- PDF-Logik in Export-Adapter
- CSV-/Excel-Import in Import-Adapter
- Statusübergänge als Fachlogik zentralisieren
- Trainer-/Azubi-Regeln zentral kapseln

--------------------------------------------------

6. Validierung vereinheitlichen
Warum:
- Input-Validierung und Business-Validierung dürfen nicht vermischt werden

Zielzustand:
- jede Route besitzt ein eigenes Input-Schema
- Validierung am Eintrittspunkt
- Fachregeln separat

Trennung:
- Input Validation:
  - Typen
  - Pflichtfelder
  - Feldlängen
  - Formate
- Business Validation:
  - darf signiert werden?
  - darf Status gewechselt werden?
  - darf User X User Y verwalten?

Nutzen:
- weniger implizite Fehler
- klarere Fehlermeldungen
- bessere Tests

--------------------------------------------------

7. Import/Export als eigene Adapter oder Use Cases schneiden
Warum:
- Import/Export wächst schnell quer durch das System
- PDF, CSV, Excel und Mapping sollten nicht in allgemeinen Services kleben

Zielzustand:
- modules/report/import/
- modules/report/export/
- modules/admin/import/
- shared/exporters/ nur für echte Generik

Empfehlung:
- Parsing
- Mapping
- Validierung
- Persistierung
sauber trennen

==================================================
PHASE 3 — FRONTEND ENTSCHÄRFEN
==================================================

8. AppContext.jsx aufteilen
Warum:
- globaler Kontext ist zu groß
- Auth, Reports, Admin, Grades, Theme und API-Aktionen laufen zu stark zusammen

Zielzustand:
- AuthProvider
- ReportProvider
- AdminProvider
- GradesProvider
- UiPreferencesProvider

Oder noch besser:
- Context nur für globalen Zustand
- Use-Case-Hooks für Aktionen und API-Orchestrierung

Empfohlene Reihenfolge:
1. AuthContext herausziehen
2. ReportContext herausziehen
3. AdminContext herausziehen
4. GradesContext herausziehen
5. Theme/Preferences separat

Praktische Regel:
Ein Context soll entweder
- globalen Zustand halten
oder
- fachliche Aktionen einer Domäne bereitstellen
aber nicht die ganze App gleichzeitig koordinieren

--------------------------------------------------

9. API-Client zentralisieren
Warum:
- Fetch-Logik sollte nicht in mehreren Contexts/Komponenten verteilt sein

Zielzustand:
- src/lib/api-client.js

Aufgaben:
- get/post/put/delete Wrapper
- Fehlerformat standardisieren
- Auth-/Session-Verhalten zentral
- optionale Hilfen für Retry bei idempotenten Requests

Nutzen:
- weniger Duplikate
- einfacheres Error Handling
- Contexts werden dünner

--------------------------------------------------

10. Container-Komponenten von UI-Komponenten trennen
Warum:
- Seiten sollten nicht gleichzeitig komplettes Laden, State-Handling und Präsentation enthalten

Zielzustand:
- pages/ = Container
- components/ = wiederverwendbare UI
- features/.../components = domänenspezifische UI

Beispiel:
- pages/TrainerDashboardPage.jsx
- features/report/components/ReportTable.jsx
- features/report/components/ReportStatusBadge.jsx

Nutzen:
- bessere Wiederverwendbarkeit
- kleinere Dateien
- leichter testbar

==================================================
PHASE 4 — ENTWICKLUNGSWORKFLOW VERBESSERN
==================================================

11. Echten Dev-Workflow einführen
Warum:
- build && node index.js für dev ist unpraktisch
- langsame Feedback-Schleifen bremsen Refactoring

Zielzustand:
- dev:server mit node --watch oder nodemon
- dev:client mit esbuild --watch
- dev zum parallelen Start

Beispiel:
- "dev:server": "node --watch index.js"
- "dev:client": "node scripts/build.mjs --watch"
- "dev": "concurrently \"npm run dev:server\" \"npm run dev:client\""

Nutzen:
- schnelleres Iterieren
- angenehmeres UI-/API-Arbeiten
- geringere Reibung bei kleinen Umbauten

--------------------------------------------------

12. Teststrategie auf Fachmodule ausrichten
Warum:
- Tests sollen die risikoreichsten Regeln absichern

Priorität:
1. Report-Statusübergänge
2. Rollen-/Berechtigungslogik
3. Import-Mapping
4. Audit-Log-Verhalten

Ziel:
- Repository-Tests
- Domain-Service-Tests
- wenige API-nahe Integrations-Tests

Regel:
- viel Kernlogik unterhalb von HTTP testen
- nur ausgewählte kritische Pfade end-to-end-nah absichern

KURZFAZIT
==================================================

Das Projekt braucht keinen Rewrite.
Es braucht eine kontrollierte Entkopplung.
Der beste Einstieg ist:
- Backend-Komposition sauber ziehen
- Frontend-Kontexte entlang der Domänen aufteilen
- API- und Validierungsschicht standardisieren

Damit bekommst du den größten Wartbarkeitsgewinn bei überschaubarem Risiko.
